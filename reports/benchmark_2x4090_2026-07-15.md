# dots.mocr на 2x RTX 4090 — тесты и бенчмарки (2026-07-15)

Автор прогона: агент Claude Fable 5.

## Окружение

| | |
|---|---|
| Хост | 2x NVIDIA GeForce RTX 4090 (24 GB), driver 565.57.01 (CUDA 12.7), 80 CPU, 251 GB RAM |
| Контейнер | `dots-mocr:bench-cu126` (`docker/Dockerfile.bench`): python 3.12-slim, torch 2.11.0+cu126, transformers 5.5.4 |
| Модель | `rednote-hilab/dots.mocr`, 3.04B параметров, bfloat16 (~6.1 GB, 2 шарда safetensors) |
| Чекпойнт | `models--rednote-hilab--dots.mocr/snapshots/main` + `scripts/prepare_checkpoint.py --in-place` |
| Документ | `mobilenetv3.pdf` (11 страниц, статья "Searching for MobileNetV3"), рендер 150 dpi (~1275x1650) |
| Промпт | `prompt_layout_all_en` (полный layout: bbox + категория + текст), greedy (temperature=0) |

Драйвер CUDA 12.7 не запускает cu13x-сборки torch (GeForce не поддерживает
forward compatibility), поэтому окружение собрано на cu126.

## Что доказано тестами

Юнит-тесты — 71 passed (локально и в контейнере на сервере): utils (smart_resize,
масштабирование bbox туда/обратно, layout→markdown, output cleaner, PDF-рендер),
парсер с подменённой моделью (артефакты, порядок страниц, температура без гонок),
веб-демо через TestClient (включая регрессию на краш `file.filename`), transformers-патч
на tiny-модели (vision-эмбеддинги действительно подмешиваются в позиции image-токенов,
`pixel_values` влияют на логиты, greedy детерминирован, save/load сохраняет веса).

GPU-интеграционные тесты — 5 passed за 179 c на реальном чекпойнте
(`tests/test_gpu_integration.py`):

- модель на cuda:0, bfloat16, >3B параметров;
- **одностраничный режим**: `prompt_ocr` по 1-й странице возвращает текст статьи,
  содержит маркер "MobileNetV3"; `prompt_layout_all_en` возвращает валидный JSON
  (>=5 ячеек, категории из словаря dots.mocr, легальные bbox), текст ячеек содержит маркер;
- `prompt_layout_only_en`: валидный JSON детекции;
- **многостраничный режим**: страницы 0-2 обрабатываются по порядку, у каждой свои
  `.md`/`.json` артефакты, содержимое страниц различается, `.jsonl` сводка пишется.

Пример выдачи модели (первые строки OCR страницы 1):

```
# Searching for MobileNetV3

Andrew Howard¹ Mark Sandler¹ Grace Chu¹ Liang-Chieh Chen¹ Bo Chen¹ Mingxing Tan² ...
```

## Сетап полной загрузки 2x4090

Модель 3B целиком помещается на одну карту (пик ~19.4 GB на страницу при 150 dpi,
включая KV-cache и vision attention), поэтому model parallel не нужен и вреден.
`benchmarks/bench_throughput.py` поднимает **по одному процессу-реплике на GPU**
(`CUDA_VISIBLE_DEVICES=i`), страницы PDF раздаются round-robin, метрики мержатся.
Утилизация обеих карт при прогоне сэмплируется `nvidia-smi` каждые 0.5 c.

## Результаты (mobilenetv3.pdf, prompt_layout_all_en, 150 dpi, max_new_tokens=8192)

| Режим | Стр. | GPU | batch | Wall, c | sec/page (система) | sec/page (латентность) | стр/мин | tokens/s (агрегат) | JSON валиден |
|---|---|---|---|---|---|---|---|---|---|
| single_page_1gpu | 1 | 1 | 1 | 46.1 | 29.1 | 29.0 | 2.06 | 43.8 | 1/1 |
| multi_page_1gpu | 11 | 1 | 1 | 506.6 | 44.5 | 44.4 | 1.35 | 47.6 | 11/11 |
| multi_page_2gpu | 11 | 2 | 1 | 274.1 | **23.4** | 45.2 | 2.57 | **90.7** | 11/11 |
| multi_page_2gpu_b2* | 11 | 2 | 2 | 165.3 | **13.9** | 50.9 | **4.31** | **150.9** | 11/11 |

\* batch=2 — с `--max-pixels 1000000`: vision-энкодер конкатенирует страницы батча в одну
последовательность, его attention квадратичен по суммарной длине; на полном разрешении
150 dpi (~2.1M px) батч из двух страниц запрашивает ~21 GB и не влезает в 24 GB.

- "sec/page (система)" = `work_seconds / pages` — пропускная способность сетапа;
  "латентность" — время `generate` для страницы (в батче страница ждёт весь батч).
- Масштабирование 1→2 GPU: **1.90x** по токенам/с (47.6 → 90.7) и по стр/мин (1.35 → 2.57).
  batch=2 добавляет ещё 1.66x (90.7 → 150.9 tok/s), суммарно **3.2x** к одной карте.
- TTFT (полный prefill страницы ~2930 токенов, 1 новый токен): **3.51 c** на обеих картах.
- Загрузка модели с диска до готовности: ~3.2 c на карту (веса в page cache).
- Память: 19.4 GB / 24 GB на карту при batch=1 @150 dpi (веса ~7 GB + KV-cache 8k +
  квадратичный vision attention); пик утилизации 100% на обеих картах, средняя ~38-41%
  в окне инференса — декод 3B-модели в HF generate ограничен латентностью запуска
  ядер (Python overhead на токен), батчинг поднимает среднюю утилизацию и throughput.
- Все 11 страниц во всех режимах дали валидный layout JSON (`valid_json_pages`).
- 100-страничный документ этим сетапом обрабатывается за ~39 мин (batch=1, полное
  разрешение) или ~23 мин (batch=2, 1.0M px).

## Исправленные баги (аудит кода предыдущего агента)

1. `demo/server.py` `/api/parse`: краш `AttributeError: 'NoneType'.filename` при вызове
   через `job_id` — основной флоу UI не работал (регрессионный тест в
   `tests/test_demo_server.py`).
2. `src/dots_mocr/cli.py`: ветка "model parallel" была мёртвым кодом (`self.device`
   резолвится в `cuda`/`cpu` до проверки `== "auto"`); параметр `--dtype` игнорировался
   (всегда bf16); `--device cpu` на GPU-машине раскладывал веса на GPU и падал на
   несовпадении устройств.
3. Гонки данных: `_parse_single_image` мутировал `self.temperature`; `parse_pdf`
   с `num_thread>1` звал `generate` на одной HF-модели из нескольких потоков —
   теперь температура передаётся per-call, `generate` под локом.
4. `demo/bench.py --max_pages` обрезал результаты после полного парсинга всех страниц.
