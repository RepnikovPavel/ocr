# 02 — Vision: flash-attention вместо sdpa (снять потолок памяти и разрешения)

> Контекст: vision-башня — не авторегрессия, а обычный encoder, живёт в prefill —
> см. [00-primer](00-primer-llm-inference-for-cuda-engineers.md) §7. Числа — [../architecture.md](../architecture.md) §3, §6.

**Одной строкой:** `cli.py` форсит `attn_implementation="sdpa"` для **обеих** башен,
из-за чего vision материализует матрицу внимания $\sim N_v\times N_v$ на голову. При
$N_v\approx11224$ это и есть **драйвер OOM** (потолок разрешения) и лишний HBM-трафик.
Flash-attention считает то же самое **без материализации** — память $O(N_v)$ вместо
$O(N_v^2)$, тензорные ядра, тайлинг on-chip.

**Важно: это не «наша идея», а восстановление авторской референс-конфигурации.**
Upstream `parser.py::_load_hf_model` грузит модель с
`attn_implementation="flash_attention_2"` **целиком** (vision + LLM), README повторяет
это в HF-примере, а requirements рекомендует `flash-attn==2.8.0.post2` («to speed up
inference need flash-attn»). sdpa-дефолт — локальное отступление нашего форка (flash-attn
не был собран в контейнере). Значит и для **LLM-части** flash — авторский референс: на
prefill это дополнительный плюс к таблице ниже, на decode влияние мало́ (там latency
решается §01).

---

## Проблема (в CUDA-терминах)

Vision-attention (полное, не каузальное, §3 архитектуры) на голову:
$$
\mathrm{softmax}\!\Big(\tfrac{q_i q_i^{T}}{\sqrt{128}}\Big)^{N_v\times N_v} \mathrm{MATMUL}\ v_i,
\qquad N_v \approx 11224
$$
Путь **sdpa/eager** явно строит буферы размером $N_v^2$:
$$
\begin{gather*}
\text{scores } h_q\times N_v\times N_v\ \text{bf16} = 12\cdot11224^2\cdot2\ \text{B} \approx 3.0\ \text{ГБ} \\
\text{softmax fp32 буфер} \approx 6.0\ \text{ГБ} \qquad (\text{+ маска } N_v^2)
\end{gather*}
$$
Именно поэтому модель на 24 ГБ живёт только при `max_pixels≈2.2M` — выше $N_v$ растёт, а
буферы растут как $N_v^2$ → OOM (это ты уже видел в бенчах). Плюс эти гигабайты
гоняются через HBM туда-обратно — чистый bandwidth-налог поверх полезного счёта.

Путь **flash** (`flash_attn_varlen_func`, уже есть в коде как ветка, но не
активирована): тайлит $Q,K,V$ по SRAM, считает softmax потоково (online-softmax), **не
пишет $N_v\times N_v$ в HBM вообще**. Для одной страницы это один varlen-сегмент
`cu_seqlens=[0,N_v]` — полное внимание по всем патчам без единого мегабайта лишних
буферов.

## Что именно делаем

1. Собрать/поставить `flash-attn` под torch+CUDA контейнера `dots-mocr:bench-cu126`
   (сборка под SM89/Ada; авторский пин — `flash-attn==2.8.0.post2`, при конфликте с
   нашим torch 2.11/cu126 собрать из исходников). В `modeling_dots_vision.py` ветка
   `VisionFlashAttention2` уже реализована — нужен только рабочий импорт
   `flash_attn_varlen_func`.
2. Вернуть авторский дефолт `attn_implementation="flash_attention_2"` в `cli.py` —
   для всей модели, как в upstream `_load_hf_model` (vision-конфиг наследует его через
   `config.vision_config.attn_implementation`); sdpa/eager оставить как fallback-опции
   CLI для машин без flash-attn.
3. Поднять `max_pixels` (сейчас искусственно зажат 2.2M из-за OOM) до реального предела
   flash — вплоть до авторского 11.29M, если позволит время prefill. Это уже не столько
   скорость, сколько **качество**: на плотных страницах (мелкий шрифт, широкие таблицы —
   ровно край из `reports/synth/`) больше патчей = выше разрешение = лучше recall.
4. Регрессия: сверить vision-эмбеддинги flash vs sdpa (bf16 tolerance) и greedy id.

## Ожидаемый uplift (честно, с выводом)

Двойной эффект — **скорость prefill** и **снятие потолка разрешения**:

**Скорость.** Vision — это $\approx60/69\approx87\%$ FLOP префилла; внутри vision
attention — $\approx54\%$ (32.5 из 60 TFLOP). Flash обычно даёт $\times1.5$–$3$ на самой
attention (за счёт отсутствия материализации и HBM-round-trip'ов) на длинных
последовательностях. Оценка по Амдалу для TTFT (доля attention в префилле
$p=32.5/69\approx0.47$, ускорение attention $s$):
$$
\text{TTFT-ускорение} = \frac{1}{(1-p) + p/s}
= \frac{1}{0.53 + 0.47/s},\qquad
s\in[1.5,\,3]\ \Rightarrow\ \times1.2\text{–}1.5
$$
Теоретический потолок (attention → бесплатно, $s\to\infty$) — $1/0.53=\times1.89$, но
реалистично $s\le3\Rightarrow$ **TTFT $3.5\,\text{с}\rightarrow\sim2.4$–$2.9\,\text{с}$**.
Больший выигрыш по TTFT здесь не заложен физикой — attention лишь половина префилла;
остальное (GEMM'ы, SwiGLU) flash не трогает.

**Разрешение/качество.** Память attention падает с $O(N_v^2)$ до $O(N_v)$ → потолок
`max_pixels` снимается. Это позволяет обрабатывать страницы, которые сейчас режутся до
2.2M px, на полном разрешении — прямой выигрыш по recall на «краевых» документах из
`reports/synth/` (широкие таблицы, мелкий шрифт). Численно uplift здесь — не «×N
скорости», а «раньше OOM / резалось → теперь считается точнее».

Что uplift **не** улучшает: decode-латентность (это §01).

## Корректность / регрессия

- flash и sdpa считают одно и то же внимание; расхождение — на уровне bf16-ULP
  (online-softmax vs полный softmax). Проверяем `allclose` на `E_img` и **точное
  совпадение greedy id**.
- Важно: flash-attn build должен матчить torch/CUDA версии контейнера, иначе тихие
  NaN/падения — прогнать `tests/test_gpu_integration.py` + пару страниц диплома до/после.

## Стоимость / зависимости / риски

- **Стоимость:** низкая по коду (ветка есть), средняя по сборке flash-attn под образ.
- **Риск:** несовпадение ABI flash-attn ↔ torch (частый источник боли); решается пином
  версий или сборкой из исходников в `docker/Dockerfile.bench`.
- **Зависимости/стыковка:** §02 — оптимизация **HF/PyTorch-пути** (наш `cli.py`,
  демки, бенчи, ступень §01-A). При переходе на vLLM (§01-B) она становится неактуальной
  для сервинга: официально интегрированный vLLM принимает картинки и исполняет
  vision-башню сам, своими attention-ядрами (без материализации $N_v^2$). HF-путь при
  этом остаётся эталоном регрессии и офлайн-инструментом — flash в нём всё равно нужен.

**Рекомендация:** делать первым или параллельно с §01-A — быстрый измеримый выигрыш по
TTFT **и** снятие OOM-ограничения, что само по себе улучшает точность.
